从零开始介绍我的 30 Keys 无拇指键 Keymap，使其顺理成章。为了简化说明，我使用普朗克（Planck）配列，因为它横平竖直（Ortholinear）。现实中我并不使用普朗克。每一节我会尝试引用一些替代方案（Alternative Options）。

最终目的：
- 30 Keys
- 无拇指键
- 不使用超过两个按键的组合
- 不放弃鼠标
- IME Numbers（CJK 用户）

我并不推荐 QMK、ZMK 键盘给任何人，无论是完全不懂键盘的新手，还是懂得键盘编程的老手。Kanata 是免费的，并且可以应用本文提到的所有技巧：<https://github.com/jtroo/kanata>。

Alternative Options：太多了。最知名的 Alternative Option 就是经典的 104 Keys（又称普通键盘、100% 键盘）。

我在这里谈论经典配列而非人们熟知的三拇指键设计（如 Corne：<https://github.com/foostan/crkbd>），和人们在后室（<https://backrooms-wiki.wikidot.com/faq#toc1>）谈论前室（<https://backrooms-wiki.wikidot.com/the-frontrooms>）的理由一样。

我当然可以继续列举一些新潮布局，但实话说，使用这些布局并没有比我在普通键盘上开心多少。我的手移动确实变少了，但我更习惯久坐了，而久坐对健康的损害是巨大的。用了四年多的分体键盘，我开始努力寻找当初反对使用普通键盘的意义。但截止到我写这篇东西的时候，我还是没有找到。对于阅读到这里的 ZMK、QMK 用户，这一节不是你们来这里的目的，对吧？

## 安排修饰键位置

简单来说，修饰键就是 Shift, Ctrl, Alt, Gui（在 Windows 上这是 Windows 键）。维基百科解释：<https://en.wikipedia.org/wiki/Modifier_key>。

如果你习惯修饰键放在角落，这没有问题，直接放在角落就行。这其中的代价就是小指疼痛，或者很难按（因为手要有某种变形才能按到像 `Ctrl + Shift + Esc` 这样的组合）。由于这是一个 30 键的键位，所以肯定要牺牲某些字母键的长按功能来作为按下修饰键。一般认为，Home Row 键（`ASDF JKL;`）是黄金地段，所以把这些键的长按功能当作修饰键。这种安排叫 **Home Row Mod**。

叙述这些是读者了解相关知识的最低门槛。欲了解更多，参考 <https://precondition.github.io/home-row-mods>。

### Home Row Mod

#### 位置安排

这里安排 ACS，从左到右是 Alt、Ctrl、Shift。右手安排是对称的，因为标准指法中，每个手指的负责区域都是固定的。Gui（我这只使用 Windows，所以是 Windows 键）稍后提及。

读者当然可以自由安排修饰键在 Home Row 的位置，根据自己在程序上的按键频率来分配。我这么安排最初是参考 Miryoku 的（<https://github.com/manna-harbour/miryoku/tree/master/docs/reference>）。我唯一需要提到的是 Ctrl。将 Ctrl 安排在 D，就可以单手按下 `Ctrl + A`（全选）、`Ctrl + S`（保存）、`Ctrl + F`（查找），它只牺牲了 `Ctrl + D`。在别的键位上安排都不能做到有如此含金量的位置。

在位置安排上，只要把 Ctrl 安排在 D，别的位置安排的 Trade-off 似乎都大差不差（或者影响较小）。

#### Home Row Mod 本身的代价

Home Row Mod 最大的 Trade-off 就是误触，其次是打字有（人为引入的）延迟。我对 QMK 了解不多，但是听说他们有这个 <https://github.com/stasmarkin/sm_td>，引进了复杂的条件判定，能够改进成完美的 Home Row Mod，使得既能流畅输入又能达到目的。对于 ZMK，使用 <https://github.com/urob/zmk-config?tab=readme-ov-file#timeless-homerow-mods>。我对 Kanata 了解更不多了。

Alternative Options：不使用 Home Row Mod 的 Keymap 可以在这里找到一些：<https://keymapdb.com/?hasHomeRowMods=false>。总的来说思路如下：

-   本文不使用拇指键，但是你可以使用拇指键来直接放置 Mod，这么做直觉上最方便。Trade-off 就是你不能用拇指键切层了。
-   创建一个 Mod Layer：<https://keymapdb.com/keymaps/callum_oakley/>。Trade-off：直觉上不方便。在普通键盘上，`Ctrl + C` 你只需要按下两个键。使用 Mod Layer 你要先切层，选 Mod，然后再按下你想要按的键，切层就多了一个键，也就是说你需要使用 `Ctrl + {nav} + C` 来按下 `Ctrl + C`。
-   使用 Combo 当作 Modifier。Trade-off：和上面一样的问题，不过平衡了一些，因为一旦习惯的话，一起按两个键在直觉上确实可以归为一个键。最大的 Trade-off 是要去习惯它，并且比上两个还难习惯。
-   使用 Leader Key（ZMK 实现：<https://github.com/urob/zmk-leader-key>）然后做 Sticky Key 或者 Key Toggle。这对 God Mode（<https://github.com/emacsorphanage/god-mode>）或者 Meow（<https://github.com/meow-edit/meow>）用户是友好的。Trade-off：基本上包含了上述所有的 Trade-off，除了 Home Row Mod 本身。

### Windows Key

刚刚说了，修饰键一共有四个，就是 Shift, Ctrl, Alt, Gui。在 Windows 上，这个 Gui 键就是 Windows 键。但是 Windows 键比较特殊，因为它事实上是被微软强制占用的键，比如 `Win + L` 锁屏，`Win + X` 执行某些东西，`Win + R` 运行。这个键很难被别的程序使用。在 Windows 上你当然可以绑定没有被微软使用的键位，但是你很难重绑定一个被使用的，比如说 `Win + L`（它甚至还使用了 Home Row 的 L）。

在思考过后，我发现我很少用到这个键。我从来没使用过 `Win + L`、`Win + X`。我用 Everything 来替换 `Win + R`。我用 Ditto 来替换 `Win + V`。我很少用 `Win + E`，并且我感觉 `explorer.exe` 较慢。我用 IME 来输入 Emoji（`Win + ;` 或者句号），IME 本身切换中英文（`Win + Space`）。于是我就将它直接踢出 Home Row 了，放在了 C 这个位置上。

### 复合修饰键（Composite Modifiers）

一次，我看到这篇文章：<https://blog.getreu.net/20250826-muscle-memory-friendly-home-row-mods>。虽然我的安排和他的不一样，但是他有一个观点很有趣：V 是 Home Row 的一种次优位置。也就是说，往下按比往上按要容易得多。自然地，我就想到要安排复合修饰键，为了达到我“不使用超过两个键”的目标。首先从 Ctrl 开始安排，由于 F 已经是 `LShift` 了，所以 V 在 F 的下面，安排 `LCtrl + LShift`。

我本可以在 X 键安排 `LCtrl + LAlt`，但是我没有，因为有更重要的东西安排了（后文的括号）。并且我使用 `LCtrl + LAlt` 较少。所以，针对难按的上半部分，安排 Alt，比如 R 键，安排 `LAlt + LShift`，然后将 E 安排了 `LCtrl + LAlt`。`LAlt` 的上方则加入了 `LMeh`（`LCtrl + LShift + LAlt`）。

## 层（Layers）

这是达到“单手鼠标”目的的又一工具，不必像前面那样小心对待，因为可以自由发挥。我经常改动层，所以我只展示基础或者不变的东西。

Alternative Option: Vim 式方向键。

方向键的安排几乎只分两个流派：Vim 流派和方向键流派。Vim 流派会把方向键横着排（像 Miryoku 将 `JKL;` 安排成左下上右，你当然也可以安排成 `HJKL`），方向键就会安排得像 `WSAD` 游戏一样。我个人不喜欢用 Vim，以及它的 Alternative Option Helix 发展不成熟，所以最终选择了方向键流派（并且也根本没有位置横着排按键）。这两个 Trade-off 的话，Vim 流派只是舍去了一点点惯性成本，但是带来的收益是上下键能更快速地按（理论上）。方向键的代价就是一根手指来控制上和下，按键压力分配不均匀。

我之前还尝试过使用旋转的方向键 Combo 来做方向键，就是 `ES` 为上，`SC` 为左，`CF` 为下，`FE` 为右。我尝试过一个月，最终结论是不值得。

### 安排数字（`num`）

前面有说过，往下按会比往上按容易得多。遵守“尽量单手”的约束，我安排了 Z 作为数字键切层（而不是 Q）。

这里的布局是 Numpad，也就是说将数字按照九宫格的布局排列（0 的位置不太重要）。Alternative Options：Numrow；Numpad 是以位置为主安排数字，而 Numrow 则是语义（从左到右依次增加）。这里有几种设计，但是都假设你在 Home Row 上使用。

-   使用平常的 Numrow，也就是普通键盘的那个。在直觉上最优，并且个人认为认知负荷较低。Trade-off：语义和频率不平衡。一般来说 1 和 0 是输入较多的数字，但是却被安排在了小拇指。不过这对于习惯的人来说无所谓。
    ![numrow](/asset/keymap/numrow.png)
-   使用频率修正的 Numrow，就像 <https://keymapdb.com/keymaps/callum_oakley/> 那样。将 1 和 0 放在食指，然后左手奇数，右手偶数。Trade-off：对比上面那个选择，它削弱了一点语义。
    ![numrow-s](/asset/keymap/numrow-s.png)

需要提醒的是我这里忽略了修饰键。实际情况中你需要考虑怎么按下 `Ctrl + 1`。不过这也很好解决：将右手的那一半 Row 放在左手 Row 的下面，因为反正往下按总比往上按好使。

-   还可以使用 Combo 输入数字，无论是竖着、横着、还是斜着。Trade-off：无法高速输入，并且习惯上，输入数字应当不比输入字母慢。需要权衡。

## 安排 Combo

有一些很重要的键，放在前文的任何一个地方似乎都不合适。所以，Combo 在这里派上用场。需要注意的是：

对称是没有意义的。左手的 Combo 镜像到右手的 Combo 对我来说语义不一致。因为我本来就主要用鼠标。当我使用右手部分的键盘时，我希望切换成“全键盘模式”。所以左手的语义和右手的语义在我看来不一致。这也就是为什么安排了 Rolling 在 X 但是没有安排在 Dot。

### Delete and Bksp

这两个是一切的开始。从 Artsey 抄来的。

![bksp-del](/asset/keymap/bksp-del.png)

### Home Row Combo

不是新东西了。不过，我需要提醒的是：Home Row 的语义在这里很重要。

`ESC`、`Tab`、`Space` 从左到右的顺序是从 Miryoku 抄来的。

![homerow-combo-left](/asset/keymap/homerow-combo-left.png)

右手的 Home Row 放符号是因为如前所述，当使用双手时，就应该打字了，此时从使用鼠标模式切换到全键盘模式。
![homerow-combo-right](/asset/keymap/homerow-combo-right.png)

### Minus, Under, Equal, Plus

使用 Minus 和 Plus 相对的语义。然后将 Combo 往上移来打出它们的变种 Under 和 Equal。同时 Equal 的下面就是 Minus，这在某些使用 `Ctrl + =` 的地方尤其有用。

![minus-plus](/asset/keymap/minus-plus.png)

### Tilde to Pipe

使用隔开的 Combo。

![to-pipe](/asset/keymap/to-pipe.png)

### IME Numbers

许多人喜欢切层输入数字，这对于不使用 IME 的人来说非常好。我相信即使使用 IME 的人也有用数字层选字的。但是我更想要一个瞬时方案。Combo 就是瞬时的，只是多按一个键。熟练以后，感觉和按一个键没什么差别。这里的 Numbers 完全没有语义，它们只是为了选字。

![num-combo](/asset/keymap/num-combo.png)

我还希望瞬时输入 Markdown 里面的 Heading 3。

![shift-num-combo](/asset/keymap/shift-num-combo.png)

对 IME Numbers 的 Combo 只有两种 Alternative Options: 
- 使用竖向的 Combo。Trade-off：会造成手指负责区域混乱，以及可能会扭到手。
- 使用斜向的 Combo。Trade-off：比数字层、横向 Combo、竖向 Combo 还难按。
