从零开始介绍我的 30 keys 无拇指键 keymap，让它变得顺理成章。为了说明更简单，我使用普朗克配列，因为它横平竖直（Ortholinear）。我现实中并不使用普朗克。每一节我会尝试引用一些 alternative options。

最终目的：
- 30 keys
- 无拇指键
- 不使用超过两个按键的行为
- 不放弃鼠标

我并不推荐 QMK、ZMK 键盘给任何人，无论是完全不懂键盘的新手，或者懂得键盘编程的老手。Kanata 是免费的，并且可以运用上这里说到的所有技巧：<https://github.com/jtroo/kanata>。

## 安排修饰键位置

简单地说，修饰键就是 shift, ctrl, alt, gui（在windows上这是windows键）。维基百科解释：<https://en.wikipedia.org/wiki/Modifier_key>。

如果你习惯修饰键放在角落，这没有问题。直接放在角落就行。这其中的代价就是小指疼痛，或者很难按（因为手要有某种变形才能按到像 Ctrl+Shift+ESC）。以及这是一个 30 键的键位，所以肯定要牺牲某些字母键的长按功能来作为按下修饰键。一般认为， homerow 键（`ASDF JKL;`）是黄金地段，所以把这些键的长按功能当作修饰键。这种安排叫 homerow mod。

叙述这些是读者了解知识的最小需要。欲了解更多，参考 <https://precondition.github.io/home-row-mods>。

### homerow mod

#### 位置安排

这里安排 ACS，从左到右是 alt、ctrl、shift。右手安排是对称的，因为标准指法中，每个手指的负责区域都是固定的。gui（在我这只使用windows，所以是 windows 键）稍后提及。

读者当然可以自由安排修饰键在homerow的位置，根据自己在程序上的按键频率来分配位置。我这么安排最初是参考 miryoku 的（<https://github.com/manna-harbour/miryoku/tree/master/docs/reference>）。我唯一需要提到的是 ctrl。将ctrl安排在d，就可以单手按下ctrl+a（全选）、ctrl+s（保存）、ctrl+f（查找），它只牺牲了ctrl+d。在别的键位上安排都不能做到有如此含金量的位置。

在位置安排上只要把 ctrl 安排在 d，别的位置安排的trade off似乎都大差不差（或者影响较小）。


#### homerow mod 本身的代价

homerow mod最大的 trade off 就是误触，第二个就是打字有（人为引入的）延迟。我对qmk了解不多，但是听说他们有这个 <https://github.com/stasmarkin/sm_td>，引进了复杂的条件判定，能够改进成完美的homerow mod使得既能流畅输入又能达到目的。对于zmk，使用<https://github.com/urob/zmk-config?tab=readme-ov-file#timeless-homerow-mods>。我对 kanata 了解更不多了。

alternative options：不使用 homerow mod 的 keymap可以在这里找到一些：<https://keymapdb.com/?hasHomeRowMods=false>。总的来说想法如下。

- 本文不使用拇指键，但是你可以使用拇指键来直接放置 mod，这么做直觉上最方便。trade off就是你不能用拇指键切层了。
- 创建一个 mod layer：<https://keymapdb.com/keymaps/callum_oakley/>。trade off：直觉上不方便。在普通键盘上，Ctrl+C你只需要按下两个键。使用mod layer你要先切层然后选 mod 然后再按下你想要按的键，切层就多了一个键，也就是说你需要使用 Ctrl+{nav}+C 来按下 Ctrl + C。
- 使用 combo 当作modifier。trade off：和上面一样的问题，不过平衡了一些，因为一旦习惯的话，一起按两个键在直觉上确实可以归为一个键。最大的trade off是要去习惯它，并且比上两个还难习惯。
- 使用 leader key（zmk实现：<https://github.com/urob/zmk-leader-key>）然后做 sticky key 或者 key toggle。这对 god mode （<https://github.com/emacsorphanage/god-mode>） 或者 meow（<https://github.com/meow-edit/meow>） 用户是友好的。trade off：基本上包含了上述所有的 trade off，除了 homerow mod 本身。

### windows key

刚刚说了，修饰键一共有四个，就是shift, ctrl, alt, gui。在windows上，这个gui键就是windows键。但是windows键比较特殊，因为它事实上是微软强行占据的一种键，比如win+L锁屏，win+x执行某些东西，win+r运行。这个键很难被别的程序使用。在windows上你当然可以绑定没有被微软使用的键位，但是你很难重绑定一个被使用的，比如说win+l（它甚至还使用了homerow的l）。

在思考过后，我发现我很少用到这个键。我从来没使用过 win+l、win+x。我用 everything 来替换 win+r。我用ditto来替换win+v。我很少用 win+e，并且我感觉 `explorer.exe` 较慢。我用 ime 来输入emoji（win+；或者句号），ime本身切换中英文（win+空格）。于是我就将它直接踢出homerow了，放在了C这个位置上。

### 复合拇指键

一次，我看到这个文章：<https://blog.getreu.net/20250826-muscle-memory-friendly-home-row-mods>。虽然我的安排和他的不一样，但是他有一个观点有趣：V是homerow的一种次优位置。也就是说，往下按比往上按要容易得多。自然地，我就想到要安排复合修饰键，为了达到我的「不使用超过两个键」的目标。首先从ctrl开始安排，由于F已经是Lshift了，所以V在F的下面，安排LC+LS。

我本可以在X键安排 LC+LA，但是我没有，因为有更重要的东西安排了（后文的括号）。并且我使用LC+LA较少。所以，针对难按的上半部分，安排alt，比如R键，安排LA+LS，然后将E安排了LC+LA。

alternative option: vim式方向键。

方向键的安排几乎只分两个流派：vim流派和方向键流派。vim流派会把方向键横着排（像miryoku将jkl;安排成左下上右，你当然也可以安排成hjkl）。方向键就会安排得像wsad游戏一样。我个人不喜欢用vim，以及它的alternative option helix发展不成熟，所以最终选择了方向键流派（并且也根本没有位置横着排按键）。这两个trade off的话，vim流派只是舍去了一点点惯性成本，但是带来的收益是上下键能更快速地按（理论上）。方向键的代价就是一根手指来控制上和下，按键压力分配不均匀。

我之前还尝试过使用旋转的方向键combo来做方向键，就是ES为上，SC为左，CF为下，FE为右。我尝试过一个月，最终结论是不值得。

这里的布局是 numpad，也就是说将数字按照九宫格的布局排列（0的位置不太重要）。alternative options：numrow；numpad是以位置为主安排数字，而numrow则是语义（从左到右依次增加）。这里有几种设计，但是都假设你在homerow上使用。

- 使用平常的 numrow，也就是普通键盘的那个。在直觉上最优，并且个人认为认知负荷较低。trade off：语义和频率不平衡。一般来说 1 和 0 是输入较多的数字，但是却被安排在了小拇指。不过这对于习惯的人来说无所谓。![numrow](/asset/keymap/numrow.png)
- 使用频率修正的numrow，就像<https://keymapdb.com/keymaps/callum_oakley/>那样。将 1 和 0 放在食指，然后左手奇数，右手偶数。trade off：对比上面那个选择，它削弱了一点语义。![numrow-s](/asset/keymap/numrow-s.png)

需要提醒的是我这里忽略了修饰键。实际情况中你需要考虑怎么按下 ctrl + 1。不过这也很好解决：将右手的那一半row放在左手row的下面，因为反正往下按总比往上按好使。

- 还可以使用 combo 输入数字，无论是竖着、横着、还是斜着。trade-off：无法高速输入，并且习惯上，输入数字应当不比输入字母慢。需要权衡。
